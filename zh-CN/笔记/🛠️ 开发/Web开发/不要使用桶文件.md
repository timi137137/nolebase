# 不要使用桶文件

## 什么是桶文件

桶文件的英文名叫`Barrel File`，能出现桶文件这个技术还是因为当年的 ECMAScript 规范缺乏模块化能力，但后期出现了ESM模块方案后，模块的数量飞速增加。模块与模块之间的依赖关系变得错综复杂。尤其是中大型项目中，通常都要跨越多层目录才能引用目标模块。例如：

```typescript
import { authService } from '../../services/auth/authService';
```

这种情况会导致迁移时路径越来越长越来越复杂，降低了代码的可读性之余，还增加了理解代码结构的难度。

### 解决方案？

因为上述情况，出现了两种解决方案，分别是 `alias` 与 `Barrel File`：

`alias` 也就是别名，通常出现在构建工具中，如TypeScript：
```json
{
  "compilerOptions": {
    "paths": {
      "@/*": ["src/*"],
      "@shared/*": ["shared/*"]
    },
  }
}
```
设置完后，导入路径就会变成：
```typescript
import { authService } from '@/services/auth/authService';
```

而`Barrel File`也就是桶文件，使用方法更是简单粗暴，在目录下创建一个`index.ts`文件：
```typescript
export * from './auth/authService';
```
引用方式直接就是：
```typescript
import { authService } from '@/services';
```

**可是？这真的就是解决方案吗？**

-----

## 真正的问题出现了

### 1. 树摇（Tree-Shaking）失效
具体原理不多哔哔，网上都能搜到。其实说白了就是算法层面没法确保桶文件里面哪些没用到哪些用到。只能全都给你保留下来。

### 2. 循环引用
这个就纯粹是要看开发人员会不会那么笨了，例如：
```text
A模块 依赖 C模块。
B模块 依赖 A模块。
C模块 依赖 B模块。
```
这种情况下正常情况是发现不了的，一个不小心就是模块未定义问题。**非常严重。很难排查！**

### 3. 影响工程化工具性能
例如最近很火的Vite工具链，如果桶文件太多的情况下，实际上你可能只用到了其中一小部分的代码。但启动开发环境的时候，工具因为树摇失效问题。要把全部导入的模块都构建，那就失去了快速开发的这么一个目的了。

----

## 最终的，解决方案
综上，虽然桶文件确实能提升开发体验，但代价却是牺牲了产物与工程环境性能，且长期来看反而会让整体模块依赖关系变得复杂难懂，我认为应该尽量克制使用 Barrel Files，使用其他方法替代，如：

1. 若项目文件结构比较简单，建议直接引用具体模块；若文件结构过于复杂，请重构，在 Monorepo 语境下做好拆包分解；
2. 如果你正在开发 NPM Package，可使用 `package.json` 的 `exports`、`typesVersion` 等字段声明导出内容，以此替代 Package 的 index 文件；

其次，应该设置一些 Lint 检测规则预防出现意料之外的 Barrel 文件，常用规则包括：

- 使用 `eslint-plugin-import` 或 oxlint 的 [`no-cycle`](https://link.juejin.cn?target=https%3A%2F%2Fgithub.com%2Fimport-js%2Feslint-plugin-import%2Fblob%2Fmain%2Fdocs%2Frules%2Fno-cycle.md "https://github.com/import-js/eslint-plugin-import/blob/main/docs/rules/no-cycle.md") 规避循环引用；
- 编写 ESLint 规则禁止 `export */import *` 一类代码，规避过度开放的 Barrel Files，不过社区似乎还没有想过实现，后续有机会再将我们内部实现的版本开源出去吧。
